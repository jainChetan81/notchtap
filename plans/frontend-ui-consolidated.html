Engineering constraints to remember

No hover exists on the real window today — mechanism now chosen (plan 086, 2026-07-21). apply_overlay_native_config (src-tauri/src/lib.rs:544-580) sets set_ignore_cursor_events(true) unconditionally on the whole overlay (:554) and stays that way — the plan 086 spike found tauri-nspanel's already-pinned, already-vendored tracking-area support delivers real mouseEntered/mouseExited/mouseMoved on this exact window with zero change to that line (empirically verified: click-through stays intact while tracking still fires). See docs/design/hover-cursor-tracking.md. The shared primitive itself SHIPPED via plan 087 (2026-07-21): hover::active_card_rect + tracking-area block + transitions-only hover-changed event + one diagnostic .hovered class. None of the four consumers (081 TTL pause / 082 weather peek / 084 scorecard reveal / idle expanded-on-hover) has wired it yet, and each must first add a height constraint — 087's rect is x-constrained only, so the full 300px window height currently counts as hovered.
Appearance preview mirror law. The Settings Appearance section renders the real StatusRailCard through src/settings/preview-overlay.css, a scoped mirror of src/styles.css. Any card change must update both files in the same commit (DESIGN.html Law #7).

Only the MacBook smoke-check remains (run with resting_state at its default "rail"). Coupled: hover.rs:27-33 mirrors these width constants in rust — any width change post-smoke-check must update both in the same commit.
